# 实习参赛记录文档设计

## 目标

在项目根目录创建一份中文 Markdown 参赛记录，作为本次 Agent
比赛的技术复盘，同时作为个人实习经历的过程记录和作品集材料。

## 文件

- 最终文件：`比赛参赛记录.md`
- 语言：中文
- 口吻：第一人称、正式、克制，兼顾技术细节与个人反思
- 读者：实习导师、面试官、项目协作者以及未来的自己

## 内容结构

1. 实习与参赛背景
2. Demo 初始架构和运行流程
3. 结合赛题暴露出的主要问题
4. 分阶段技术改进
5. 测试、诊断和工程化验证
6. 当前遗留问题与风险
7. 个人承担的工作与能力成长
8. 对实习和比赛的总结

## 赛题问题

文档将基于仓库证据讨论以下问题：

- 题目输入结构与公开字段处理不够稳健。
- 多文件审计容易产生大量逐文件调用和证据串线。
- 顺序 API 测试容易错误复用 token、打乱请求顺序或遗漏断言。
- 日期自然语言计算存在边界错误。
- Java 修复题需要正确识别类名、编码、参数和多用例执行。
- 原生工具调用失败后的 JSON 回退范围过宽，可能重复执行任务。
- answer checker 可能越权修改语义，而不是只修复格式。
- 运行期间结果文件、错误位置和调用过程不透明。
- 工具输出和上下文过长会增加模型负担。
- 总题目超时前预留 90 秒生成兜底答案，但兜底请求本身若超时，
  当前实现仍会写入空答案。

## 改进记录

文档将按实际提交和设计材料概括以下改进：

- 输入契约、结果预分配和原子写入。
- 生命周期诊断、Trace 和可选本地 Dashboard。
- 模型网关、流式响应、超时和输出预算配置。
- 工具调用并行化与依赖顺序控制。
- 子 Agent、answer checker 和格式保护。
- 日期计算、压缩包安全解压、Java 执行和 package ID 传播。
- `dataset_bundle_read`、`evidence_chain_analyze` 和
  `api_test_execute` 等确定性工具。
- Java 个税题专用 Skill 与任务提示。
- 最大迭代次数和软截止时间控制。

## 事实边界

- 不写未提供的公司、学校、导师或具体岗位名称。
- 不虚构比赛名次、得分、获奖情况或正式上线效果。
- 已完成的能力和测试使用确定语气。
- 未完成的端到端验证、环境限制和遗留缺陷使用明确的限制说明。
- Remotion `Hello World` 视频属于后续项目展示实验，不作为比赛核心能力。

## 验证

完成最终文档后：

- 对照核心代码、设计文档和 Git 提交核查事实。
- 检查时间线、配置默认值和术语是否一致。
- 搜索空占位符和未经证实的成绩表述。
- 检查 Markdown 标题层级、链接和代码标识。
