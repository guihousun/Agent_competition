# 难度 1：证据链定位与接口测试设计

## 目标

增强 Agent 对两类难度 1 任务的稳定性：

- `1_3`：从截图、前端日志、HAR、后端校验日志和规则文件中定位唯一完整前台作业流，并输出模块、接口和根因。
- `1_4`：根据接口文档、鉴权配置和自然语言测试描述，按顺序执行全部用例并输出失败用例 ID。

实现不得依赖题号、公开样本答案、固定业务模块、固定接口路径或固定关联字段。

## 方案比较

### 方案 A：继续依赖主 Agent 逐文件读取和逐请求判断

改动最小，但会产生大量 LLM 和工具调用。模型容易跨作业流拼接证据，也容易漏掉接口字段或错误复用 token。历史运行已经证明结果不稳定。

### 方案 B：为公开样本编写完整特化求解器

速度快，但会固化文件名、接口和业务字段，隐藏题变化后容易失效，不符合泛化要求。

### 方案 C：通用结构化执行工具，Agent 负责语义规划

推荐该方案。工具负责解析、关联、请求执行和机械断言；Agent 负责选择入口、解释自然语言动作以及生成最终答案。这样减少模型调用，同时不绑定公开答案。

## 总体架构

新增两个 MCP 工具：

1. `evidence_chain_analyze`
2. `api_test_execute`

两个工具都只接收题目声明的文件路径，并经过现有 `LocalMCPClient` 路径权限校验。工具输出结构化摘要，不直接写比赛最终答案。

主 Agent 的系统提示仅增加一条短规则：遇到对应输入组合时优先使用批量分析工具。不得加入公开题的具体模块、接口、测试 ID 或答案。

## `1_3` 证据链分析

### 输入

- 一个或多个日志文件
- HAR 文件
- 规则 JSON
- 可选截图或截图压缩包

工具参数使用文件角色而非固定文件名：

```json
{
  "frontend_paths": ["..."],
  "backend_paths": ["..."],
  "har_paths": ["..."],
  "schema_path": "...",
  "screenshot_paths": ["..."]
}
```

### 规范化

工具把 JSON、JSON Lines、键值日志和 HAR 记录规范化为事件。每个事件保留：

- 来源和原始位置
- 时间字段
- 请求方法和接口路径
- HTTP 状态
- 页面或模块字段
- 可用关联标识，例如 `workflowId`、`actionId`、`requestId`、`traceId`、`validationRef`
- 校验码、阶段、来源、字段路径和 schema 版本
- 前台人工操作、后台任务、诊断、回放或预校验等语义标签

字段名不做单一硬编码。工具递归提取标量字段，并通过同名值、请求上下文和时间邻近关系建立候选边。

### 作业流选择

每个候选作业流必须尽量覆盖：

1. 页面或截图记录
2. 前端人工提交事件
3. 网络请求
4. 后端校验
5. 最终可见失败

候选链按覆盖完整度、强关联标识数量、时间一致性评分。诊断、回放、预校验、静态资源和后台任务降低评分。工具不把两个存在冲突标识的事件合并。

若最高分候选不唯一，工具返回 `ambiguous: true` 和候选摘要，让主 Agent 继续核验，而不是猜测。

### 规则应用

工具动态识别规则文件中的三种契约：

- `validationCodeMap`
- `effectiveValidationRule`
- `rootCauseRules`

先确定作业流，再应用规则。规则匹配支持嵌套字段路径和标量相等、集合包含等基础条件。输出同一作业流中全部有效根因，并保留规则命中证据。

### 输出

```json
{
  "ambiguous": false,
  "selected_flow": {
    "module": "...",
    "endpoint": "...",
    "root_causes": ["..."],
    "event_count": 5
  },
  "evidence": [],
  "rejected_candidates": []
}
```

主 Agent 只负责将该结构转换为题目要求的字符串。

## `1_4` 顺序接口测试

### 两阶段执行

自然语言描述可能变化，因此不在工具中硬编码动作解析规则。

第一阶段由主 Agent 一次性读取接口文档、鉴权配置和所有测试用例，将全部用例转换成结构化执行计划。每个步骤包含：

- HTTP 方法、URL 或路径
- headers、query、body
- 是否获取新 token
- 是否复用指定 token
- 是否从响应中提取变量
- 本用例最终要断言的步骤

第二阶段由 `api_test_execute` 一次调用顺序执行整个计划。

### 会话和状态

- 全部请求使用运行时同一个 `X-Package-Id`。
- 严格保持用例和步骤顺序。
- token 可从配置指定的 JSON 路径提取。
- 默认不自动复用一次性写 token；只有计划明确要求时才复用。
- 前序写入产生的服务状态保留给后续用例。
- 不自动调用 reset，除非执行计划明确包含该动作。

### 断言

工具提供确定性断言：

- HTTP 状态相等
- JSON 字段存在
- JSON 路径值相等
- `null`、布尔值、数字和字符串严格比较
- 数组下标路径，例如 `data.list.99.userId`

单个用例的网络或解析错误记为该用例失败，但不终止后续用例。输出保持原始用例顺序。

### 输出

```json
{
  "failed_case_ids": ["TC001"],
  "passed_count": 19,
  "failed_count": 1,
  "cases": [
    {
      "id": "TC001",
      "passed": false,
      "assertion_errors": ["..."],
      "steps_executed": 3
    }
  ]
}
```

工具不包含公开测试 ID 或期望值。所有断言来自输入执行计划。

## 错误处理

- 文件非法或缺失：返回结构化 `error`，不抛出不可读堆栈。
- HAR 或日志部分损坏：保留可解析记录，并在 `warnings` 中说明。
- 接口返回非 JSON：仍记录状态和文本，JSON 字段断言失败。
- 请求超时：当前用例失败，继续后续用例。
- 执行计划非法：在发出任何请求前完成校验并拒绝执行，避免半执行状态。
- 证据链不唯一：显式返回歧义，不虚构确定结论。

## 测试策略

先写失败测试，再实现。

### `1_3`

- 通过动态 `traceId` 或 `requestId` 串起完整前台链路。
- 排除诊断、回放、预校验和后台任务。
- 不跨冲突工作流合并根因。
- 分别覆盖三种规则契约。
- 最高分链路并列时返回歧义。
- 损坏日志行不影响其他记录。

### `1_4`

- 严格顺序执行用例和步骤。
- 所有请求继承同一个 package ID。
- 从 token 响应路径提取并格式化鉴权头。
- 一次性写 token 按计划刷新或复用。
- 校验嵌套字段、数组下标和 `null`。
- 单用例失败后继续执行后续用例。
- 计划校验失败时不发送请求。

最后运行当前完整单元测试，并在本地服务可用时用官方 `1_4` 输入做端到端验证。`1_3` 因当前发布目录未包含附件，使用合成变体测试验证泛化能力；获得真实附件后再补端到端回归。

## 不在本次范围

- 不修改 `2_x` 题目能力。
- 不增加新的语义 checker。
- 不把题目路由绑定到题号。
- 不在主 Prompt 中加入公开答案或详细 few-shot。
- 不改变比赛结果写入和日志机制。
